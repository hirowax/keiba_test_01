### 目的
現在 `pickup_scores.json` には `today_pop`（前日人気）しか保存されておらず、オッズがないため期待値計算ができない。スコアリング時点（前日夜〜当日朝）の**単勝オッズ**を保存し、将来のEVフィルター検証の材料を貯める。

### 変更箇所

**1. `race_pickup.py` の `scrape_shutuba()`**（69行目〜）

- 人気取得（125行目付近、`td.Popular` / `td.Ninki` を探している箇所）と同じ `tr` 内から単勝オッズも取得する
- netkeiba shutuba.html のオッズは通常 `<td class="Txt_R Popular"><span id="odds-XX_YY">n.n</span></td>` の形式。**実装時に必ずライブHTMLで実セレクタを確認すること**（`page.content()` をファイル保存して目視）

- 173行目の speed.html フォールバック側でもオッズが取れるなら同様に取得。取れなければ `odds_map` は空でよい（欠損許容）

**2. `run_pickup_all.py`**（319行目付近、`pop_map` を保存している箇所）

**3. 互換性**

- `rescore.py` / `rerun_failed_pickup.py`: `odds_map` がない過去日付の pickup_scores.json を読んでもエラーにならないこと（`.get()` で参照）

## Phase B: キャリブレーション + EVバックテスト（2026-11頃・クリーン90日到達後）

### 事前登録する意思決定ルール（実装前にこの定義で固定。データを見て調整しない）

```
買い判定: EV = p(3着内 | スコア) × 複勝オッズ推定値 ≥ 1.1 のとき「買い」、未満は「見送り」
```

- **p(3着内 | スコア)**: 直近クリーン窓のスコア別3着内率。スコアビンは `5-6 / 7 / 8 / 9+` の4区分、各ビンにLaplace平滑化（+1/+2）を適用
- **複勝オッズ推定値**: 優先順位順に
  1. Phase Aで保存した単勝オッズがあれば「人気→複勝オッズ中央値テーブル」を単勝オッズ由来の人気で引く
  2. なければ `today_pop` でテーブルを引く
  3. どちらもなければ判定不能（買い対象外・集計からも除外）

### 人気→複勝オッズ中央値テーブル（2026-07-05に80日分・約1.6万頭で作成済み）

| 確定人気 | N | 3着内率 | 複勝オッズ中央値 |
|---|---|---|---|
| 1 | 1186 | 67.6% | 1.30 |
| 2 | 1189 | 52.2% | 1.50 |
| 3 | 1183 | 41.0% | 1.90 |
| 4 | 1185 | 34.8% | 2.20 |
| 5 | 1189 | 26.2% | 2.60 |
| 6 | 1185 | 21.6% | 3.40 |
| 7 | 1181 | 15.3% | 4.20 |
| 8 | 1163 | 12.8% | 4.70 |
| 9 | 1126 | 9.9% | 5.70 |
| 10+ | 5964 | 4.2% | 5.90 |

※市場の複勝期待値は人気1で0.88、以降単調減少（テラ銭約20〜25%）。◎のスコア情報がこのベースラインを複勝オッズ換算で+15%以上押し上げるレースだけが「買い」になる設計。
※Phase B実施時は最新データでテーブルを再作成すること（`race_results.json` の `pop` × `place_odds` から集計）。

### バックテスト評価（判定基準も事前固定）

- 窓: クリーン窓全体 / 直近30日 / それ以前、の3分割で一貫性を確認
- 指標: **複勝ROIを主指標**とする（単勝ROIは外れ値で歪むため参考値扱い。`docs/anaba_research_log.md` の教訓）
- 外れ値チェック: 的中上位3本の払戻を除外したROIを必ず併記
- **採用基準（すべて満たしたときのみPhase Cへ）**:
  - 買い対象の複勝ROI ≥ 100%（外れ値除外後も ≥ 85%）
  - 買い対象の3着内率 ≥ 55%
  - 買い対象N ≥ 40
  - 全3分割窓で複勝ROIが見送り対象を上回る
- 基準未達の場合は「保留」としてこのドキュメントに結果を追記し、さらにデータ蓄積を待つ。**閾値1.1を動かして再テストしない**（多重検定）

### 既知のバイアス（レポートに明記すること)
- 過去分の複勝オッズテーブルは「確定人気」ベース。前日人気とは乖離がある（前日人気の馬が当日人気を落とすとオッズは上がる方向）。Phase Aで貯めた実測前日オッズが90日分揃えば、この近似は不要になる
- `today_pop` 欠損（◎の約30%）問題: shutuba人気の取得率改善は別途検討（`himo_analysis_202607.md` §6.1 データ品質メモ参照）

## Phase C: 運用反映（Phase B採用基準クリア後のみ）

### 表示方式: 損益分岐オッズ表示（2026-07-06 設計決定）

前日→当日のオッズ変動に対応するため、「最新オッズをページに取り込む」のではなく
**「変動しない判定基準（損益分岐オッズ）をページに表示し、当日オッズとの比較は人間が行う」**方式を採用する。

- 根拠1: サイトは静的GitHub Pagesであり、ブラウザから外部オッズをfetchするのはCORS制限で不可能。
  リロードボタン等でのオッズ更新はローカルスクレイプ→push（1〜2分+cookies負荷）が必要になり割に合わない
- 根拠2: JRAはパリミュチュエル方式のため払戻は締切時オッズで確定する。「買った時点のオッズ」は固定されないので、
  どの時点のオッズで判定しても最終オッズの近似。判定式のうち p(3着内|スコア) は前日に確定するため、
  **損益分岐オッズ = 1.1 / p** は前日に計算でき、当日動かない
- 実測（2026-07-06・1,134頭）: 前日人気と確定人気は完全一致70.7%・±1以内84.7%、
  人気帯（1-3番人気 vs 4番人気以下）の維持率は89〜92% → 前日情報での帯レベル判定は十分安定

### 実装内容

- `pickup_scores.json` の各馬に `ev_place`（数値）と `bet_flag`（buy/skip）と `breakeven_odds`（損益分岐複勝オッズ = 1.1/p を小数1桁）を追加
- `index.html`: 🔥馬のカードに「複勝◯.◯倍以上なら買い」を表示 + そのレースのnetkeibaオッズページへのリンクを設置
  （当日はワンタップでオッズを開き、分岐オッズと見比べるだけの運用）
- 見送りレース（EV判定不能・分岐オッズが非現実的に高い等）はトーンダウン表示。**期待ROIの数値は表示しない**（anaba研究の原則）
- `calibrate_threshold.py` はそのまま（◎選定の的中率目標とEVフィルターは独立に保つ）

# 柱3: JRA払戻パイプライン + 3連複の事前登録検証

## Step 1: JRA公式払戻スクレイパー `scrape_jra_payouts.py`

### 背景
馬連・ワイド検証（himo_analysis_202607.md §5）はnetkeibaの払戻を使ったが、ユーザー方針として**レース結果・払戻はJRA公式（jra.go.jp）から取得する**。3連複検証にも全レースの払戻が必要。

### 判明済みの事実（2026-07-05にライブ検証済み）

1. **URL**: `https://www.jra.go.jp/JRADB/accessS.html?CNAME={cname}` （GET可・**ログイン不要**・Shift_JIS）
2. **cname構造**（レース結果ページ）: `pw01sde{XX}{VV}{YYYY}{KK}{DD}{RR}{YYYYMMDD}/{CS}`
   - `XX`: ページ種別（`01`=当週系, `10`=過去系。srl経由で取れたリンクをそのまま使えば意識不要）
   - `VV`=場コード（netkeibaと同一: 札幌01 函館02 福島03 新潟04 東京05 中山06 中京07 京都08 阪神09 小倉10）
   - `KK`=回次, `DD`=日目, `RR`=レース番号, 末尾に開催日8桁
   - `CS`: 2桁16進チェックサム。**計算式は未解明（位置重み付き和 mod 256 系だが完全特定に至らず）。計算せず、ページ内リンクから収集する**。誤ったCSは「パラメータエラー」ページになる（title で判定可能）
   - netkeiba race_id `YYYYVVKKDDRR` と成分が1対1対応するため、目的レースのsdeリンクを特定できる
3. **クロール経路**（チェックサム付きリンクの収集方法）:
   - 今週の成績一覧: POST `cname=pw01skl00999999/B3` → 月別ページへのリンク `pw01skl10{YYYYMM}/{CS}` が拾える
   - 月別ページ: 前月等へのリンク（BFSで過去月に到達可能）+ 日別レース一覧リンク `pw01srl10{VV}{YYYY}{KKDD}{YYYYMMDD}/{CS}`
   - 日別ページ(srl): 全レースのsdeリンク（12R分）が載っている
4. **払戻HTML構造**: `<li class="umaren">` / `<li class="wide">`(要実確認) / `<li class="trio">`(3連複) / `<li class="tierce">`(3連単) 内に `<div class="num">2-12</div>` `<div class="yen">3,930<span>円</span></div>`。ワイドは複数行

### 実装仕様

```
python3 scrape_jra_payouts.py YYYYMMDD    # 1日分
python3 scrape_jra_payouts.py --all       # race_results.jsonがあり payouts_jra.json がない全日付
```

- **出力**: `output/{date}/payouts_jra.json`
  ```json
  {"函館10R": {"tansho": {"5": 180}, "fukusho": {"5": 110, "4": 160, "8": 170},
               "umaren": {"4-5": 660}, "wide": {"4-5": 240, "5-8": 260, "4-8": 480},
               "umatan": {"5-4": 900}, "trio": {"4-5-8": 1080}, "tierce": {"5-4-8": 3150}}}
  ```
  組合せキーは馬番昇順（馬単・3連単は着順のまま）
- **リンクキャッシュ**: `output/jra_cname_cache.json` に月別/srl/sdeのcname（チェックサム込み）を保存し、再実行時のクロールを省略
- **マナー**: リクエスト間 1〜2秒スリープ、UAはChrome、1回の実行での取得はガイドとして最大3日分（=最大108レース+ナビページ）
- **文字コード**: `decode('shift_jis', errors='replace')`
- レース名・場名の対応: srlページ内の場名표示 or cnameのVVから場コード変換

### 実装ステップ
1. **プローブ**: 月別ページを202510までBFSで遡れるか確認。届かなければ到達できた最古月を記録し、それ以前は「JRA公式で取得不可」としてこのドキュメントに追記（その範囲の扱いはユーザーに確認）
2. 1日分（例: 20260628）で実装・実行し、下記の照合で検証
3. `--all` で全日付を複数回に分けて取得

### 検証（netkeiba払戻との照合）
2026-07-05にnetkeibaから取得済みの値と一致すること（例）:
- 20260404 阪神10R 馬連 900円 / ワイド 320円
- 20260125 中山6R ワイド 1,230円
- 20260328 中京9R ワイド 1,380円

## Step 2: 3連複の事前登録検証（Step 1完了後すぐ実行可能）

### 検証仮説（事前固定・変更禁止）
「3着内率63%の◎を軸に、紐を広め（B∪D∪C）に流す3連複は、馬連・ワイド（検証済みNG）より市場の歪みが残っており回収率が高い」

### 買い目定義（himo_analysis_202607.md §3-4 の定義を再利用）
- 対象レース: ◎（スコア1位かつ8pt以上）が存在するレース
- 軸: ◎
- 紐: B（◎以外の3指数重複馬）∪ D（単指数1位・非重複）∪ C（2指数重複・非重複）。平均約3.7頭
- 買い目: ◎-紐2頭の3連複 C(k,2)点（k=紐頭数。k<2のレースは見送り）
- 1点100円

### 参考数値（既知データからの見込み）
- ◎好走レースのうち紐2頭以上好走: 25%（17/69・クリーン窓） → レース単位的中率見込み ≈ 16%
- 平均約6.3点/R（k≈3.7） → 損益分岐は平均払戻 約4,000円

### 評価（事前固定）
- 窓: クリーン窓 / 全期間 の2窓 + 月別推移
- 指標: 回収率、的中率、**的中上位3本除外後の回収率**（外れ値チェック必須）
- **「有望」判定基準**: 素の回収率 ≥ 110% かつ 上位3本除外後 ≥ 70% かつ 的中数 ≥ 15
- 基準未達なら結果をこのドキュメントに追記して終了。**紐の範囲（B∪DのみやC∪Eなど）を変えて再テストしない**（多重検定。やるなら次の90日データで1回だけ）
- ついでに馬連・ワイドもJRA払戻で再集計し、netkeiba集計値（馬連27.4%/ワイド67.0%）との整合を確認
